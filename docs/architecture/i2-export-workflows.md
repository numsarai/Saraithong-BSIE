# i2 Export Workflows

This note documents the two IBM i2 Analyst's Notebook export paths that BSIE now produces from the shared graph bundle.

## Why There Are Two Outputs

BSIE generates:

- `i2_chart.anx`
- `i2_import_spec.ximp`
- `i2_import_transactions.csv`

These are not duplicates.

They serve different analyst workflows:

- `i2_chart.anx`
  - open a ready-made chart directly in Analyst's Notebook
  - best when the analyst wants the fastest path from BSIE to a visible chart
- `i2_import_spec.ximp` + `i2_import_transactions.csv`
  - import through the Analyst's Notebook import wizard or `SeriesImport.exe`
  - best when the analyst wants i2 to create cards/attributes through the import engine, keep an import specification, and re-run the same import pattern later

Both outputs come from the same deterministic `nodes.csv` and `edges.csv` graph bundle.

## Direct Chart Path: `.anx`

The direct chart export is generated by [`core/export_anx.py`](../../core/export_anx.py).

Design intent:

- keep the chart readable
- exclude aggregate rows
- exclude bookkeeping-only lineage edges
- preserve match-review safety semantics

Included edge types:

- `OWNS`
- `SENT_TO`
- `RECEIVED_FROM`
- `MATCHED_TO`
- `POSSIBLE_SAME_AS`

Key characteristics:

- stable node IDs are preserved
- analyst-friendly descriptions are emitted onto chart items
- node positions are deterministic enough for reproducible charts
- flow links keep chart datetime metadata when dates are parseable

Use this path when:

- you want to open BSIE output immediately as a chart
- you do not need the import wizard
- you want a compact visual surface instead of a denser transaction import

## Import Package Path: `.ximp + .csv`

The import package is generated by [`core/export_i2_import.py`](../../core/export_i2_import.py).

Design intent:

- stay close to i2's built-in financial-transaction import examples
- map one transaction row into a small, legible analyst shape
- preserve evidence lineage as import attributes

The current import package focuses on transaction-flow edges only:

- `SENT_TO`
- `RECEIVED_FROM`

That narrower scope is deliberate. It avoids flooding the import workflow with bookkeeping-only or review-only edges that are better represented in the direct chart or the raw graph CSVs.

### CSV Contract

The companion CSV includes:

- source and target account identity
- labels and account numbers
- owner and bank fields
- transaction date/time/amount/currency
- reference number and description
- review/assertion/confidence fields
- transaction ID and source transaction IDs
- file ID, parser run ID, statement batch ID
- source file, source sheet, and source row number

This lets analysts keep evidentiary pointers attached to imported transaction links inside i2.

### Specification Shape

The `.ximp` specification:

- defines two `Account` entities
- defines a transaction link between them
- maps date and time fields for the transaction card
- maps lineage and transaction metadata into link attributes
- keeps a layout block so imported items are charted predictably

## Analyst Usage

### Option 1: Open the ready-made chart

1. Generate or download `i2_chart.anx`
2. Open it in Analyst's Notebook
3. Use this when you want the quickest review path

### Option 2: Import through i2

1. Keep `i2_import_spec.ximp` and `i2_import_transactions.csv` in the same folder
2. In Analyst's Notebook, import the specification
3. Let the specification resolve the companion CSV path

For command-line import, Analyst's Notebook 9 exposes `SeriesImport.exe`.

Observed CLI usage from the installed Windows utility:

```text
SeriesImport -b ImportSpec [-w] [-i InputFile] [-p TemplateFile] [-l LogFile] [-f] [-c]
```

Practical usage for BSIE artifacts:

```powershell
& "C:\Program Files (x86)\i2 Analyst's Notebook 9\Utils\SeriesImport.exe" `
  -b "C:\path\to\i2_import_spec.ximp" `
  -i "C:\path\to\i2_import_transactions.csv" `
  -l "C:\path\to\series-import.log"
```

If the CSV stays beside the `.ximp`, the specification already points at the companion file.

## Verification Performed

The export work was verified against real Analyst's Notebook 9 assets, not just by eyeballing XML:

- the i2 importer specification schema from the installed product was inspected
- the built-in financial transaction `.ximp` example was compared against the BSIE mapping design
- the generated `.ximp` was validated on Windows with the .NET XML schema validator against the real importer XSD
- the generated `.anx` was validated on Windows with the .NET XML schema validator against the real Analyst's Notebook chart XSD tree

That validation step was done on Windows because older i2 XSD files use constructs that macOS `xmllint` does not compile reliably.

## Regenerating Existing Artifacts

To refresh historical output directories after exporter changes, use:

```bash
./.venv/bin/python /Users/saraithong/Documents/bsie/scripts/regenerate_i2_exports.py
```

This scans:

- `/Users/saraithong/Documents/bsie/data/output`
- `/Users/saraithong/Documents/bsie/data/exports`

and rewrites:

- `i2_chart.anx`
- `i2_import_transactions.csv`
- `i2_import_spec.ximp`

For per-account output packages it also updates `meta.json` so the new i2 files appear in `category_files`.

## Current Boundaries

- direct opening of `.anx` via Parallels/macOS shell association was unreliable in command-line automation, so schema validation was used as the trustworthy compatibility check
- the import package currently models transaction flow edges, not the full review/match surface
- richer graph semantics still live in `nodes.csv`, `edges.csv`, `aggregated_edges.csv`, and the graph analysis outputs
