Assessment & Recommendations

GOOD Schema design follows investigation-grade patterns: append-only evidence, full lineage tracking, confidence scoring, duplicate detection, and audit trails.
GOOD Proper indexing on high-cardinality columns (transaction_fingerprint, normalized_account_number, file_hash_sha256) and composite index on (account_id, transaction_datetime).
GOOD UUID primary keys enable safe distributed inserts and cross-system references. NUMERIC(18,2) for amounts and NUMERIC(5,4) for confidence scores provide sufficient precision.
GOOD WAL journal mode + foreign keys enabled. Raw evidence preserved in raw_import_rows for reproducibility.
WARN account_entity_links has 34,970 records but no unique constraint on (account_id, entity_id, link_type). This may cause duplicate links. Consider adding a UNIQUE constraint.
WARN transaction_matches has 73,752 records (1.8x transactions). This is a high ratio. Consider periodic cleanup of low-confidence matches (status="suggested", confidence < 0.3) to reduce DB size.
WARN No CASCADE DELETE rules on any foreign key. Deleting a file won't clean up its parser_runs, raw_import_rows, or transactions. This is intentional for evidence preservation but makes reset operations expensive.
WARN Legacy tables duplicate functionality: mapping_profile vs mapping_profiles, job vs parser_runs. Consider migrating legacy data and dropping the old tables when safe.
INFO Database size (382 MB) is manageable for SQLite with WAL mode. Performance will remain good up to ~500K transactions. Beyond that, consider PostgreSQL migration.
INFO JSON columns (lineage_json, raw_row_json, metadata_json) add flexibility but are not queryable by index. If you need to search by JSON fields, consider extracting frequently-queried values into proper columns.
INFO Missing index suggestion: Add index on transactions.posted_date for date-range queries which are common in investigation workflows.
